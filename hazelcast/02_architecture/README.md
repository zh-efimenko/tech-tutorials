# Урок 2. Архитектура: Embedded vs Client-Server

## Две топологии Hazelcast

Hazelcast поддерживает два режима работы: **embedded** (встроенный) и **client-server**. Выбор топологии — одно из первых архитектурных решений при внедрении Hazelcast.

## Embedded-топология

В embedded-режиме каждый экземпляр приложения является **полноценным членом кластера** Hazelcast. Данные хранятся в той же JVM, что и бизнес-логика.

```
┌────────────────────────────────────────────┐
│                  Cluster                   │
│                                            │
│  ┌──────────────┐    ┌──────────────┐      │
│  │   App + HZ   │◄──►│   App + HZ   │      │
│  │   Member 1   │    │   Member 2   │      │
│  │              │    │              │      │
│  │ Data Part.   │    │ Data Part.   │      │
│  │ 0-135        │    │ 136-270      │      │
│  └──────────────┘    └──────────────┘      │
└────────────────────────────────────────────┘
```

**Преимущества:**

- Минимальная латентность — данные в той же JVM, нет сетевых вызовов для локальных партиций
- Простота деплоя — не нужен отдельный кластер, достаточно запустить приложение
- Меньше инфраструктуры — нет отдельных серверов для управления

**Недостатки:**

- Деплой приложения = перебалансировка кластера (repartitioning)
- GC приложения влияет на Hazelcast и наоборот
- Масштабирование данных привязано к масштабированию приложения
- Rolling update сложнее: каждый рестарт пода — это выход и вход члена кластера

**Когда использовать:**

- Небольшие сервисы с 2-3 инстансами
- Прототипы и MVP
- Когда латентность критична (микросекунды vs миллисекунды)
- Локальная разработка и тесты

## Client-Server топология

В client-server режиме Hazelcast-серверы работают **отдельно** от приложений. Приложения подключаются как клиенты — они не хранят данные, а только отправляют операции.

```
┌─────────────────────────────────────────────┐
│              Hazelcast Cluster               │
│                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ HZ       │  │ HZ       │  │ HZ       │  │
│  │ Server 1 │◄─►│ Server 2 │◄─►│ Server 3 │  │
│  │ Data     │  │ Data     │  │ Data     │  │
│  └────▲─────┘  └────▲─────┘  └────▲─────┘  │
│       │              │              │        │
└───────┼──────────────┼──────────────┼────────┘
        │              │              │
   ┌────┴────┐   ┌────┴────┐   ┌────┴────┐
   │  App 1  │   │  App 2  │   │  App 3  │
   │(client) │   │(client) │   │(client) │
   └─────────┘   └─────────┘   └─────────┘
```

**Преимущества:**

- **Независимое масштабирование** — серверы Hazelcast и приложения масштабируются отдельно
- **Стабильность кластера** — деплой приложения не вызывает repartitioning
- **Изоляция памяти** — GC приложения не влияет на Hazelcast
- **Разделение ответственности** — данные управляются отдельно от логики

**Недостатки:**

- Дополнительная сетевая задержка на каждую операцию (обычно 0.5-2 мс)
- Сложнее инфраструктура — нужно управлять отдельным кластером
- Требуется мониторинг подключений клиентов

**Когда использовать:**

- Production-среды с частыми деплоями
- Микросервисная архитектура
- Kubernetes — серверы в StatefulSet, приложения в Deployment
- Когда данные должны пережить полный рестарт всех инстансов приложения

## Сравнение топологий

| Характеристика | Embedded | Client-Server |
|---------------|----------|---------------|
| Латентность (локальная партиция) | ~1-10 мкс | ~0.5-2 мс |
| Деплой приложения | Repartitioning | Без влияния на данные |
| Масштабирование данных | Вместе с приложением | Независимое |
| Количество JVM | N (приложение = член) | N серверов + M клиентов |
| GC-изоляция | Нет | Да |
| Сложность инфраструктуры | Низкая | Средняя |
| Production-готовность | Для малых кластеров | Рекомендуемый подход |

## Партиционирование

Независимо от топологии, данные в Hazelcast распределяются по **партициям**. По умолчанию — 271 партиция.

### Как работает партиционирование

```
IMap.put("user:42", userData)
        │
        ▼
hash("user:42") = 8923741
        │
        ▼
8923741 % 271 = partition 156
        │
        ▼
Partition 156 → Node 2 (owner)
               → Node 3 (backup)
```

1. Ключ хэшируется (MurmurHash3)
2. Хэш делится по модулю на количество партиций
3. Каждая партиция назначена конкретному узлу (owner)
4. Backup-партиции находятся на других узлах

### Перебалансировка при изменении кластера

Когда узел добавляется или удаляется, Hazelcast автоматически перераспределяет партиции. Это происходит **инкрементально** — перемещаются только те партиции, которые нужно перебалансировать, а не все данные.

```
До: 2 узла                    После: добавлен 3-й узел
Node1: Part 0-135              Node1: Part 0-90
Node2: Part 136-270            Node2: Part 91-180
                               Node3: Part 181-270
                               (Партиции 91-135 мигрировали с Node1)
                               (Партиции 181-270 мигрировали с Node2)
```

### Partition-aware операции

Hazelcast-клиент знает, на каком узле находится каждая партиция (**smart routing**). Операция `get(key)` отправляется **напрямую** на узел-владелец, без промежуточных шагов.

```java
// Клиент знает: ключ "user:42" → partition 156 → Node 2
// Запрос идёт напрямую на Node 2, минуя Node 1 и Node 3
map.get("user:42");
```

> **Важно:** Smart routing работает по умолчанию. Если отключить его (`smart-routing: false`), все запросы пойдут через один узел — это резко снизит производительность.

## Структура файлов конфигурации

В client-server топологии нужны **две** конфигурации:

```
project/
├── docker/hazelcast/
│   ├── hazelcast.yaml           # Серверная конфигурация
│   └── hazelcast-local.yaml     # Серверная конфигурация (локальная разработка)
│
└── src/main/resources/
    ├── hazelcast-client.yaml       # Клиентская конфигурация (production)
    └── hazelcast-client-local.yaml # Клиентская конфигурация (локальная разработка)
```

**Серверная конфигурация** определяет:
- Имя кластера
- Настройки сети и discovery
- Конфигурацию карт (бэкапы, индексы, eviction)
- Split-brain protection
- Метрики

**Клиентская конфигурация** определяет:
- Имя кластера (должно совпадать с серверным)
- Адреса серверов
- Стратегию подключения и переподключения
- Near Cache (опционально)
- Сериализацию

## Discovery-механизмы

Hazelcast поддерживает несколько способов обнаружения членов кластера:

| Механизм | Где используется | Конфигурация |
|----------|-----------------|--------------|
| **Multicast** | Локальная сеть (по умолчанию) | `multicast.enabled: true` |
| **TCP-IP** | Известные адреса серверов | `tcp-ip.enabled: true` + список адресов |
| **Kubernetes** | K8s-кластер | `kubernetes.enabled: true` + service DNS |
| **AWS** | Amazon EC2 | `aws.enabled: true` + IAM role |
| **GCP** | Google Cloud | `gcp.enabled: true` + service account |
| **Azure** | Microsoft Azure | `azure.enabled: true` + credentials |

Пример TCP-IP discovery:

```yaml
hazelcast:
  cluster-name: my-cluster
  network:
    join:
      multicast:
        enabled: false
      tcp-ip:
        enabled: true
        member-list:
          - 192.168.1.10
          - 192.168.1.11
          - 192.168.1.12
```

Пример Kubernetes discovery:

```yaml
hazelcast:
  network:
    join:
      kubernetes:
        enabled: true
        service-dns: hazelcast.default.svc.cluster.local
```

## Практика

1. Запусти два Hazelcast-сервера в Docker с одним `cluster-name` и TCP-IP discovery, убедись что они образовали кластер (в логах будет `Members {size:2}`)
2. Напиши клиентское приложение, которое подключается к кластеру и кладёт 1000 записей в IMap
3. Останови один сервер и убедись, что данные по-прежнему доступны с клиента
4. Запусти embedded-режим: создай `Config` и `Hazelcast.newHazelcastInstance(config)` в приложении, положи данные и прочитай
5. Сравни латентность embedded vs client-server: замерь время 10 000 операций `put` и `get` в обоих режимах
6. Запусти третий сервер и проверь в логах, что произошла перебалансировка партиций

## Итоги урока

- Hazelcast поддерживает две топологии: embedded (приложение = член кластера) и client-server (серверы отдельно)
- Client-server — рекомендуемый подход для production: независимое масштабирование, стабильность при деплоях, GC-изоляция
- Embedded подходит для малых кластеров, прототипов и тестов, даёт минимальную латентность
- Данные разделяются на 271 партицию, каждая имеет owner и configurable backups
- Smart routing позволяет клиенту отправлять запросы напрямую на узел-владелец данных
- При добавлении/удалении узлов партиции перебалансируются инкрементально
- Discovery-механизмы: multicast, TCP-IP, Kubernetes, облачные провайдеры
- В client-server топологии нужны две конфигурации: серверная и клиентская
