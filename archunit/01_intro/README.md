# Урок 1. Что такое ArchUnit и зачем он нужен

## Проблема: архитектура деградирует молча

В каждом сервисе есть договорённости о структуре: контроллеры не обращаются к репозиторию напрямую, сущности не зависят от сервисов, утилиты не тянут бизнес-логику. Пока команда маленькая — всё держится на ревью и дисциплине.

Со временем появляются исключения: "один раз можно", "потом рефакторим", "deadline горит". Каждое такое исключение — сигнал, что следующему разработчику тоже можно. Архитектура деградирует не взрывом, а постепенно, и это почти невозможно отследить ревью.

## Что такое ArchUnit

**ArchUnit** — библиотека для автоматической проверки архитектурных правил Java/Kotlin кода. Правила описываются как обычные JUnit-тесты, которые запускаются вместе с остальными тестами в CI.

Вместо "договорились на ревью не использовать Hibernate напрямую в контроллерах" — написал тест, и нарушение упадёт на `./gradlew test`.

ArchUnit работает на уровне байткода: загружает `.class`-файлы и анализирует их структуру — зависимости между классами, аннотации, пакеты, иерархии наследования. Исходники не нужны.

## Как это выглядит на практике

```java
// Нарушение архитектуры — как обычный упавший тест
@ArchTest
static final ArchRule controllers_should_not_use_repositories =
    noClasses()
        .that().resideInAPackage("..controller..")
        .should().dependOnClassesThat()
        .resideInAPackage("..repository..");
```

Если кто-то из команды сделает `@Autowired BetRepository betRepository` в `BetController` — этот тест упадёт с чётким сообщением:

```
Architecture Violation [Priority: MEDIUM] - Rule 'no classes that reside in a package '..controller..'
should depend on classes that reside in a package '..repository..'':
  Class <com.example.controller.BetController> has a field of type
  <com.example.repository.BetRepository> in (BetController.java:12)
```

## Что ArchUnit умеет проверять

| Категория | Примеры |
|-----------|---------|
| **Зависимости** | Кто может зависеть от кого, запрет конкретных импортов |
| **Слои** | Layered architecture, onion, hexagonal |
| **Именование** | Классы в пакете `service` должны оканчиваться на `Service` |
| **Аннотации** | Все `@RestController` должны иметь `@Validated` |
| **Наследование** | `JpaRepository` реализуют только классы в пакете `repository` |
| **Циклы** | Нет цикличных зависимостей между пакетами |
| **Видимость** | Сервисы не имеют публичных полей |

## Место ArchUnit в тестовой пирамиде

```
         /\
        /  \   E2E тесты
       /----\
      /      \  Интеграционные тесты
     /--------\
    /          \ Unit тесты
   /------------\
  /              \ Архитектурные тесты (ArchUnit)
 /________________\
```

Архитектурные тесты — самый нижний уровень. Они не проверяют бизнес-логику, но гарантируют, что структура кода соответствует договорённостям команды. Стоят дёшево: не поднимают Spring-контекст, не ходят в БД, работают быстро.

## Альтернативы и отличия

| Инструмент | Подход | Когда |
|------------|--------|-------|
| **ArchUnit** | Тесты в коде, JUnit, байткод | Большинство случаев |
| **Checkstyle** | XML-конфиг, статический анализ | Форматирование, стиль кода |
| **PMD / SpotBugs** | Статический анализ | Баги, уязвимости |
| **JDepend** | Метрики зависимостей | Только анализ, не правила |

ArchUnit уникален тем, что правила — это код. Их можно переиспользовать, наследовать, параметризировать. Документация к правилу — это сам тест.

## Практика

1. Открой любой сервис в проекте и найди хотя бы одну неявную архитектурную договорённость (например, "контроллеры только вызывают сервисы")
2. Посмотри на структуру пакетов сервиса — сколько слоёв, как они называются
3. Подумай, какое нарушение проще всего совершить случайно — именно с этого стоит начать писать правила

## Итоги урока

- ArchUnit — библиотека для проверки архитектурных правил через JUnit-тесты на уровне байткода
- Правила описываются как обычный Java-код, запускаются вместе с unit-тестами в CI
- При нарушении тест падает с точным сообщением: какой класс, какая зависимость, какая строка
- ArchUnit проверяет зависимости, слои, именование, аннотации, наследование и цикличность
- Архитектурные тесты не поднимают Spring-контекст — они быстрые и дешёвые
- Альтернативы (Checkstyle, PMD) решают другие задачи — ArchUnit специфичен для архитектурных ограничений
