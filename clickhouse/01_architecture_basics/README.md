# Урок 1. Архитектура и основы ClickHouse

## Что такое ClickHouse

ClickHouse — колоночная аналитическая СУБД (OLAP), разработанная в Яндексе. Предназначена для выполнения аналитических запросов над большими объёмами данных в реальном времени.

## Колоночное vs строковое хранение

### Строковое (PostgreSQL, MySQL)

Данные хранятся построчно. Запрос `SELECT avg(price) FROM orders` читает **все** колонки каждой строки, хотя нужна только `price`.

```
| id | user_id | price | created_at |
|  1 |     100 |  9.99 | 2026-01-01 |  ← читается вся строка
|  2 |     101 | 19.99 | 2026-01-02 |  ← читается вся строка
```

### Колоночное (ClickHouse)

Каждая колонка хранится отдельно. Тот же запрос читает **только** колонку `price`.

```
id:         [1, 2, 3, ...]
user_id:    [100, 101, 102, ...]
price:      [9.99, 19.99, 29.99, ...]   ← читается только это
created_at: [2026-01-01, 2026-01-02, ...]
```

**Выгода**: в аналитике обычно нужны 3-5 колонок из 50+. Колоночное хранение читает в 10-100x меньше данных с диска.

## Ключевые свойства ClickHouse

| Свойство | Описание |
|----------|----------|
| **Скорость** | Миллиарды строк в секунду на одном сервере |
| **Сжатие** | Данные одного типа в колонке сжимаются в 5-20x |
| **SQL** | Стандартный SQL с расширениями |
| **Append-only** | Оптимизирован под вставку, а не обновление отдельных строк |
| **Линейное масштабирование** | Кластер из N серверов — N-кратный рост производительности |

## Когда использовать ClickHouse

**Подходит:**
- Аналитические дашборды (агрегации, фильтры, группировки)
- Логи и события (clickstream, метрики, трейсы)
- Временные ряды (мониторинг, IoT)
- Ad-hoc аналитика по большим данным

**Не подходит:**
- OLTP (частые UPDATE/DELETE отдельных строк)
- Транзакционная обработка
- Key-value хранение
- Поиск по тексту (для этого — Elasticsearch)

## Архитектура

```
┌─────────────────────────────────────────┐
│              ClickHouse Server          │
│                                         │
│  ┌──────────┐  ┌──────────┐            │
│  │  HTTP     │  │  Native  │  ← Порты  │
│  │  :8123    │  │  :9000   │            │
│  └──────────┘  └──────────┘            │
│                                         │
│  ┌─────────────────────────────────┐   │
│  │        Query Processor          │   │
│  │  (парсинг → план → выполнение)  │   │
│  └─────────────────────────────────┘   │
│                                         │
│  ┌─────────────────────────────────┐   │
│  │      Storage Engine             │   │
│  │  MergeTree (основной движок)    │   │
│  │  - Parts (куски данных)         │   │
│  │  - Merges (фоновые слияния)     │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

### Два протокола доступа

- **HTTP (порт 8123)** — используется из приложений, curl, Grafana. Простой текстовый протокол.
- **Native TCP (порт 9000)** — используется `clickhouse-client` и JDBC-драйвером. Бинарный, быстрее.

## Практика

### 1. Запуск ClickHouse

```bash
docker compose up -d
```

### 2. Подключение через CLI

```bash
docker exec -it clickhouse-local clickhouse-client --user default --password clickhouse
```

### 3. Первые запросы

```sql
-- Версия сервера
SELECT version();

-- Список баз данных
SHOW DATABASES;

-- Создание базы для обучения
CREATE DATABASE IF NOT EXISTS tutorial;

-- Переключение на базу
USE tutorial;

-- Системная информация
SELECT * FROM system.metrics LIMIT 10;
```

### 4. HTTP-интерфейс

```bash
# Через curl
curl 'http://localhost:8123/?user=default&password=clickhouse&query=SELECT+version()'

# С форматированием в JSON
curl 'http://localhost:8123/?user=default&password=clickhouse&query=SELECT+1+as+id,+2+as+value+FORMAT+JSON'
```

## Ключевые отличия от PostgreSQL/MySQL

| Аспект | PostgreSQL | ClickHouse |
|--------|-----------|------------|
| Хранение | Строковое | Колоночное |
| UPDATE | Быстрый, строка за строкой | Медленный (перезапись куска) |
| INSERT | Строка за строкой | Батчами (1000+ строк за раз) |
| Агрегации | Медленно на больших объёмах | Очень быстро |
| Транзакции | Полный ACID | Нет полноценных транзакций |
| Индексы | B-tree, Hash, GIN, GiST | Sparse index (разреженный) |
| Типичный объём | Гигабайты | Терабайты и петабайты |

## Итоги урока

- ClickHouse — колоночная OLAP-СУБД для аналитики
- Хранит данные по колонкам → читает только нужные данные
- Два протокола: HTTP (:8123) и Native TCP (:9000)
- Оптимизирован под INSERT батчами и SELECT с агрегациями
- Не замена PostgreSQL, а дополнение для аналитического слоя
